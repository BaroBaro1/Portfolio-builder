# 14 - UI & Design System

# Portfolio Builder

## UI & Design System

Version: 1.0

Status: Draft

---

# Purpose

This document defines every visual rule of the application.

Every screen, component and interaction must follow these rules.

Consistency is more important than creativity.

---

# Product Personality

Portfolio Builder is not a portfolio website.

It is a platform for building a professional identity.

The interface must feel:

* Modern
* Calm
* Professional
* Clean
* Trustworthy
* Fast
* Minimal

The user should feel:

> "This platform helps me build my professional future."

---

# Design Philosophy

The UI must prioritize:

* Simplicity
* Clarity
* Readability
* Consistency
* Accessibility
* Speed

Never decorate for the sake of decoration.

Every element must have a purpose.

---

# Visual Inspiration

Inspired by the design philosophy of:

* Linear
* Vercel
* Notion
* Stripe
* GitHub

Not copied.

Only inspired by their consistency and simplicity.

---

# Color System

## Primary

Emerald

Represents:

* Growth
* Success
* Professionalism
* Confidence

---

## Neutral Palette

Slate

Used for:

* Text
* Borders
* Backgrounds
* Cards

---

## Accent

White

Used to keep the interface clean.

---

## Semantic Colors

Success → Green

Warning → Amber

Error → Red

Information → Blue

Never invent additional semantic colors.

---

# Theme Support

Version 1 supports:

* Light Theme
* Dark Theme

Every component must support both.

Dark Mode is never optional.

---

# Typography

Primary Font:

Geist

Fallback:

sans-serif

Rules:

* Large titles are bold.
* Paragraphs remain lightweight.
* Maximum readability.
* Never use decorative fonts.

---

# Typography Scale

Hero

Heading 1

Heading 2

Heading 3

Body Large

Body

Small

Caption

The same scale must be used throughout the project.

---

# Spacing System

Base Unit:

8px

All spacing must be multiples of 8 whenever possible.

Avoid arbitrary spacing.

---

# Border Radius

Small

Medium

Large

Default radius:

12px

Never mix different visual styles.

---

# Shadows

Shadows must remain subtle.

Cards should appear elevated, not floating.

Heavy shadows are forbidden.

---

# Borders

Borders are preferred over strong shadows.

Use borders to separate content.

---

# Containers

Maximum content width:

7XL Container

Content must never stretch across the entire screen.

Large empty margins improve readability.

---

# Layout Rules

Desktop First Quality

Mobile First Implementation

Every screen must work perfectly on:

* Mobile
* Tablet
* Laptop
* Desktop

Responsive behavior is mandatory.

---

# Navigation

Navbar

Clean.

Minimal.

Sticky.

Contains only essential actions.

No unnecessary links.

---

Sidebar

Visible only inside Studio.

Never on the public website.

---

Footer

Minimal.

Professional.

Useful.

No visual clutter.

---

# Buttons

Primary

Filled Emerald

Secondary

Outline

Ghost

Minimal

Danger

Red

Button sizes remain consistent across the project.

---

# Inputs

Simple.

Readable.

Accessible.

Strong focus state.

Large click area.

Never use tiny inputs.

---

# Cards

Rounded

Bordered

Minimal Shadow

Consistent Padding

No decorative gradients.

---

# Icons

Library:

Lucide

Never mix icon libraries.

Icons always have consistent size.

---

# Motion

Animations exist only to improve UX.

Never for decoration.

Duration:

Fast

Smooth

Natural

Animations include:

* Fade
* Scale
* Slide

Nothing else unless necessary.

---

# Loading States

Every asynchronous action must display feedback.

Allowed:

* Skeleton
* Spinner
* Progress

Never leave users wondering.

---

# Empty States

Every empty page must guide the user.

Example:

"No projects yet"

Instead of:

Blank page.

---

# Accessibility

Keyboard navigation supported.

Visible focus.

Readable contrast.

Large touch targets.

Semantic HTML.

Accessibility is a requirement.

Not an improvement.

---

# Copywriting Style

Short.

Friendly.

Professional.

Action-oriented.

Prefer:

"Create Portfolio"

Instead of:

"Submit"

Buttons must describe the action.

---

# Component Rules

Every component must be:

Reusable

Independent

Responsive

Theme-aware

Accessible

Documented

No duplicated UI.

---

# Design Consistency

Before creating a new component ask:

* Can an existing component solve this?
* Does it respect the Design System?
* Does it improve consistency?

If the answer is no,

Do not build it.

---

# Future Components

Every future UI component must follow this document.

The Design System has priority over personal preference.

---

# Golden Rule

If a design decision makes the product look cleaner,

simpler,

and easier to use,

it is usually the correct decision.

If it only makes the interface look "cool",

reject it.
